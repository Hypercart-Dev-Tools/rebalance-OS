PROJECT OVERVIEW
rebalance-OS is a local-first second brain over work artifacts, centered on an Obsidian vault, SQLite/sqlite-vec, and MCP/CLI interfaces.
The current push is to make GitHub artifact ingestion and readiness inference useful enough for sprint planning and deployment goal-setting, while feeding weekly summaries back into the vault so they become searchable next-week knowledge.

PROJECT ANTI-GOALS (OPTIONAL)
1. Do not rely on live-skimming hundreds of GitHub artifacts at answer time.
2. Do not move core knowledge workflows to cloud-hosted storage or embedding services.
3. Do not hide project-state reasoning behind opaque inference with no evidence trail.
4. Do not spend time on cosmetic planning workflows before local data contracts are stable.

A. STRATEGIC GOALS
1. Make the Obsidian vault, GitHub artifacts, and generated weekly reviews act as one local knowledge system.
2. Build a trustworthy GitHub corpus in SQLite with both exact metadata and semantic retrieval.
3. Turn repo signals into explicit readiness states with confidence, blockers, and evidence for sprint/deploy planning.
4. Preserve local-first portability and security: no hidden cloud dependency, no secret leakage, and no adapter-specific logic in the core.

C. LAST WEEKS ACCOMPLISHMENTS
1. Added local GitHub artifact sync into SQLite for issues, PRs, comments, reviews, commits, labels, milestones, releases, repo metadata, and branches.
2. Added semantic GitHub retrieval over the local artifact corpus plus MCP/CLI access for querying it.
3. Implemented explicit GitHub release-readiness inference with inspectable status, blockers, evidence, and confidence.
4. Added weekly note write-back so end-of-week summaries become vault artifacts that can be re-ingested and embedded automatically, plus follow-up repo hygiene around local DB sidecar ignores.

D. RECENT LESSONS LEARNED
1. Semantic retrieval is useful, but deploy planning still depends on first-class structured GitHub metadata.
2. Explicit evidence and blockers are more trustworthy than “smart” hidden reasoning for project-state answers.
3. Generated artifacts become more valuable when they flow back into the vault and local vector store instead of living as throwaway exports.
4. Better repo hygiene helps, but the system still needs resilient heuristics because real repos are never perfectly labeled or linked, and reminder/task systems can add the “human follow-up” signal GitHub misses.
